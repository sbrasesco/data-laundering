## ✅ Lo que se hizo hoy

### FIX-PDF-JOBS-CREATE (COMPLETADO Y EN PROD)

**Problema:** `pdf_jobs` no se insertaba para jobs de `supabase_storage`. La UI mostraba vacío aunque el pipeline procesara todo correctamente (outputs depositados, créditos cobrados, archivos movidos a `procesados/`).

**Root cause:** `gateway.mjs` hacía `POST /rest/v1/pdf_jobs?on_conflict=id` con `catch (_) {}`. `fetch` no lanza en 4xx/5xx — el error se tragaba silenciosamente.

**Fix:**
- Migración `add_gateway_create_pdf_job_rpc` aplicada en Supabase:
  ```sql
  CREATE OR REPLACE FUNCTION public.gateway_create_pdf_job(
    p_job_id uuid, p_organization_id uuid, p_status text DEFAULT 'processing',
    p_client_id uuid DEFAULT NULL, p_input_source text DEFAULT 'supabase_storage'
  ) RETURNS void LANGUAGE plpgsql SECURITY DEFINER SET search_path TO 'public' AS $$
  BEGIN
    INSERT INTO pdf_jobs (id, organization_id, status, client_id, input_source)
    VALUES (p_job_id, p_organization_id, p_status, p_client_id, p_input_source)
    ON CONFLICT (id) DO NOTHING;
  END; $$;
  ```
- `gateway.mjs`: reemplazado REST INSERT por llamada a esta RPC + logging estructurado (`gateway.pdf_job_created` / `gateway.pdf_job_create_failed`).
- Worker redeployado y confirmado: pipeline supabase_storage e2e funciona, `pdf_jobs` se crea correctamente.
